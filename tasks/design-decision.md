# Design: Task 22 — Productize native multi-model agents (install + auto-boot lane)

## Context

Two verified-but-dev-machine-bound setups run GPT-5.5 (Codex) and Gemini as first-class subagent types inside Claude Code, via a local relay on port 3456 that routes on the request body's `model` field (`claude-*` → api.anthropic.com verbatim; everything else → VibeProxy :8317 with Claude credentials stripped). The task is to productize this into an installable playbook surface: one skill that (a) checks-and-guides the machine layer (VibeProxy app + OAuth — never automated), (b) writes the project/user layer (relay, agent type files `codex` / `codex-xhigh` / `gemini-flash`, settings wiring), (c) wires auto-boot so the relay is running before any relayed session — the "env var set but relay down" total lockout is the central design driver — and (d) ends with a per-lane doctor probe (agent self-report + relay-log upstream assertion).

Shipped Gemini lane is VibeProxy OAuth only (zero GCP); the Vertex/ADC lane stays in-repo but toggled off and unoffered. The OAuth lane ships untested by design — the doctor probe is each user's install-time verification and must fail loudly and diagnosably (relay down / toggle off or not logged in / model ID unsupported).

Hard constraints (from research): hardened relay non-negotiable (abort nets + credential strip on all non-Anthropic routes); one relay per port 3456 machine-wide; relay never a child of a claude session; stock sessions stay 100% stock and plain `claude` must never be locked out; templates project-agnostic; every shipped file enumerated in `/playbook-update`'s managed list (`.claude/settings*` can never be managed — the install skill writes settings); agent-type names frozen for task 23.

**Research:** `tasks/research-codebase.md`

## Options Considered

All three options share a spine of axis choices that coupling or constraints effectively force; they differ on **Axis 5 (auto-boot mechanism) + Axis 6 (env opt-in boundary)** — the lockout axes — and the knock-ons that follow.

### Shared spine (all options)

- **Axis 1 (a) — one skill, two modes:** `/native-agents` (name proposal, developer-overridable) with bare/`install` = install flow, `doctor` = probe. The restart-before-doctor boundary splits the phases temporally regardless, so the surface is naming, not flow. Precedent: `checkpoint/SKILL.md:12` multi-mode `$ARGUMENTS`.
- **Axis 2 (c) — split shipped placement:** boot/doctor helper script(s) under `.claude/scripts/native-agents/` (live path, but inert without settings wiring — hooks only fire if settings reference them, and settings are never shipped), while the relay, the three agent `.md` files, and the Vertex machinery ship as inert templates under `.claude/templates/native-agents/`. Coupling forces inertness for agent files: committed live `.claude/agents/*.md` would register agent types in every consuming repo pre-install, violating "stock sessions stay stock" (research, Axis Coupling). Keeping the boot script at a managed live path means relay-boot fixes auto-propagate via `/playbook-update`.
- **Axis 3 (b) — one canonical relay:** merge the ~10-line delta (`relay-gemini.mjs:33-37,70-73` vs `relay.mjs:30-32,64-67`): gemini upstream defaults to 8317 (VibeProxy = shipped OAuth lane), env-var/flag flips to 8319 (Vertex, local-only). Strip condition uses the broadened `!== ANTHROPIC` form. Kills the two-relays-on-3456 footgun by construction and collapses Axis 9.
- **Axis 4 (b) — machine-global installed home:** install copies the relay to `~/.claude/native-agents/` (relay + `relay.log`). One process, one log path, one version machine-wide; per-project copies would race on 3456 with version skew (research, Axis Coupling). Mirrors the `/playbook-setup` Step 3B copy-to-`~/.claude/` precedent with diff-and-confirm on re-run (`playbook-setup/SKILL.md:100-122`). Resolves research OQ6: the doctor's log path is always `~/.claude/native-agents/relay.log`.
- **Axis 7 (c)+(a) — layered doctor in the user's fresh session:** install ends "start a fresh relayed session, then run `/native-agents doctor`". Doctor = step 0: verify the doctor's own session is relayed (`ANTHROPIC_BASE_URL` points at 3456) — if not, instruct relaunching via the chosen opt-in mechanism → preflight (curl relay health + VibeProxy `/v1/models`; identify what's bound to 3456) → spawn each installed agent type in-session (agent types registered at session start — hence the restart) with a self-report prompt, gemini probe includes a tool-use step (tool path untested) → grep `relay.log` for the matching upstream + any requested-vs-served mismatch lines. Headless `claude -p` probing (choice b) is proven viable (`tasks/gemini-native-agents.md` §4) but stays a documented fallback — it sharpens the issue-#3 permission problem for no added signal. Resolves research OQ3: the dev test scripts stay dev-only; the doctor uses curl + in-session agent spawns.
- **Axis 8 (c) — live-catalog machine checks:** assert against live `/v1/models` (gpt-* present → codex lane OK; gemini-* present → Gemini toggle on), with app-presence (`/Applications/VibeProxy.app`) as the diagnostic fallback when the endpoint is down. Hardcoded catalogs drift (VibeProxy README already stale vs served models — research, External Research).
- **Axis 9 (b) — hidden toggle on the canonical relay:** Vertex lane = env var/flag (e.g. `RELAY_GEMINI_UPSTREAM=8319`) documented only in template comments, never in user-facing text. Follows from Axis 3 (b).
- **Axis 10 (c) — documented model-ID checklist + live-catalog doctor:** frontmatter `model:` can't be single-sourced without a generation step, and install-time placeholder filling (choice b) would force generated installed files (Axis 11 b). Ship current IDs (`gpt-5.5`, `gemini-3.5-flash`) verbatim, enumerate every drift site in the skill's update note, and let the doctor's live `/v1/models` assertion detect drift at run time.
- **Axis 11 (a) — verbatim copies:** installed files are byte-copies of managed templates; update path = `/playbook-update` (templates refresh) → re-run `/native-agents install` (diff-and-confirm propagates to live paths, Step 3B pattern). This keeps the relay's security-fix channel intact: managed template updates are visible, and re-install diffs flag the change.
- **Axis 12 (b) — install writes its own permission rules:** merge the rules the doctor and boot path need (`node`, `curl`, `lsof`, the boot script) into the consuming project's local settings during the project-layer write. Issue #3 (general fresh-install permission gap) stays open and unowned by this task; this feature just stops depending on it.

**Settings-file refinement (applies to Axes 5/6/12 in every option):** any machine-coupled wiring the install writes (permission rules, and — in the env-in-settings options — env/hooks) goes in project `.claude/settings.local.json` (per-machine, gitignored), NEVER the shared `.claude/settings.json`. The shared file is committed: writing `ANTHROPIC_BASE_URL` there would relay every teammate's sessions on machines that never ran the machine-layer install — the worst-case lockout, inflicted on people who never opted in. **Codex cross-check CORRECTION (verified against `tasks/todo.md:407`):** even `settings.local.json` env fails the constraint's strict reading — "a user who skips the install **(or launches plain `claude`)** sees zero behavior change … a broken/absent relay must never lock anyone out of plain sessions." Env in any settings file makes plain `claude` in the installed repo a relayed session; the verified design's own framing agrees ("the relay only matters to sessions launched with `ANTHROPIC_BASE_URL`" — `tasks/codex-native-agents.md:117`). This reframes Options 1 and 3: they survive only under a looser reading of 407 ("stock" = non-installed users/projects). The agent `.md` files in project `.claude/agents/` remain shareable (official behavior); anyone in a non-relayed session who spawns `codex` gets a contained per-spawn model error, not a locked session — the skill documents this, and the shipped agent descriptions can name the relayed-session requirement.

### Option 1 — Hook-booted, project-local relayed (Axis 5 = C, Axis 6 = a-local)

**How it works:** install writes to `.claude/settings.local.json`: `env.ANTHROPIC_BASE_URL = http://localhost:3456`, plus a SessionStart hook (correct `"hooks"` nesting — `tasks/errors.md:5`) invoking `.claude/scripts/native-agents/boot-relay.sh`. The boot script: if 3456 is listening → exit 0; else start `~/.claude/native-agents/relay.mjs` detached (setsid/nohup — survives the session, re-parented to init, satisfying "never a claude child"), poll until listening (~2s cap), exit 0 unconditionally (fail-open). Every session in this repo on this machine is relayed and self-heals at session start (startup/resume/clear/compact all fire SessionStart); other projects and machines are untouched.

**Good:** agent types work in *every* session in the installed project — no half-state where agents are registered but routing is broken. Lazy boot covers reboots without daemon machinery. Strongest pattern fit: task 18's SessionStart research is already in the repo (`todo.md:349-390`), hook pitfalls documented (`tasks/errors.md:5`), settings-write precedent in `/playbook-setup`. Fully reversible: delete the settings.local entries and the machine is stock.

**Not good:** lockout protection concentrates in hook correctness — if the hook fails to fire (trust/approval gate on externally-edited hooks, wrong nesting) *and* the relay is down, sessions in this repo break until the user removes the env line (escape hatch must be printed by install and doctor). A mid-session relay crash isn't healed until the next SessionStart event.

**Timing verified (2026-06-11, dev machine):** Claude Code waits for SessionStart hook completion before the first model API call. Experiment: hook sleeping 5s + stub server on the base URL logging arrivals — first `POST /v1/messages` landed 49ms *after* hook-end (hook-end `…245883` → first POST `…245932`). Headless `claude -p` fires SessionStart hooks too, and the CLI auto-retries failed API calls with backoff (extra cushion). One residual: a `HEAD /` connectivity probe fires *before* the hook — not a model call, so a cold boot may log a transient connectivity warning at most. Research OQ5 is resolved in Option 1's favor on timing.

**However (Codex cross-check, verified):** the option fails the strict reading of `todo.md:407` — env in `settings.local.json` means plain `claude` in this repo is no longer stock, and a hook failure + dead relay locks plain sessions out, which 407 says "must never" happen. Option 1 is rejected on this constraint, not on timing — if 407 is ever relaxed, this lane is revival-ready (timing evidence above stands, hook-approval flow would be the remaining open item).

### Option 2 — Installed fail-closed launcher (Axis 5 = A refined, Axis 6 = b)

**How it works:** install copies a real launcher script to the machine home — `~/.claude/native-agents/bin/claude-native` (name proposal; Codex cross-check TRADE-OFF, absorbed: an installed file follows the Step 3B copy/diff precedent far better than asking users to edit shell startup files). The launcher: if 3456 isn't listening → start `~/.claude/native-agents/relay.mjs` detached (setsid/nohup — re-parented to init before exec, satisfying "never a claude child"), poll until listening (~2s cap); **if the relay still isn't verifiably listening, refuse to start Claude with a diagnostic** (fail-closed — env is never set against a dead relay, making the env-set-but-relay-down lockout impossible by construction); else `exec env ANTHROPIC_BASE_URL=http://localhost:3456 claude "$@"`. No settings env, no hooks. Shell integration is one check-and-guide line (PATH entry or alias) the user pastes themselves. Headless flows work identically (`claude-native -p …`).

**Good:** the only option that satisfies `todo.md:407` strictly — plain `claude` is byte-for-byte stock everywhere, installed repo included, and lockout is eliminated rather than mitigated. Least machinery: no hooks, no daemon, no settings writes beyond permission rules. Relay boot is still automatic (AC5: the launcher boots it — "auto-boot" attaches to the relay process, and the spec's own framing ties relaying to launch-time env, `codex-native-agents.md:117`). Matches the dev machine's verified working pattern (the `claude-codex()` zsh wrapper, codex doc §5A) — this is productization of the proven shape, not a new mechanism.

**Not good:** per-session opt-in is a habit change — users must launch relayed sessions via `claude-native`; forgetting it yields the half-state where project `.claude/agents/` registers `codex`/`gemini-flash` but spawning them errors (contained per-spawn, and the doctor's step-0 relayed-session check plus a "requires relayed session" line in the shipped agent descriptions make it diagnosable — but it will happen). A mid-session relay crash isn't healed by anything (no supervisor; the CLI's API retry-with-backoff is the only cushion). PATH/alias wiring is check-and-guide, shell-dependent, and unverifiable by the skill until the doctor runs.

### Option 3 — launchd-supervised + project-local env (Axis 5 = B, Axis 6 = a-local)

**How it works:** same settings.local env as Option 1, but instead of a hook, install renders a LaunchAgent plist (stable absolute paths into `~/.claude/native-agents/`), `launchctl bootstrap`, `RunAtLoad` + `KeepAlive`. Coupled requirement: relay template must flip `uncaughtException` to fail-fast under supervision (`relay.mjs:134-135`, `todo.md:406`) — keep-serving + a wedged process is exactly what KeepAlive can't fix.

**Good:** strongest relay-up guarantee, including mid-session crash recovery (the one case Option 1 can't heal). The usual "macOS-only" objection is moot — VibeProxy is a macOS 13+ menu-bar app, so the entire feature is macOS-bound regardless.

**Not good:** most moving parts and zero playbook precedent for daemon management: plist lifecycle (bootstrap/bootout on update + uninstall), launchctl permission prompts on fresh machines (worsens the Axis 12 surface), stale-plist-vs-updated-relay skew, fail-fast + KeepAlive restart-loop flapping when misconfigured. Hardest option to diagnose for users who don't know launchd. Least reversible (machine-level artifact outliving the repo). The settings-env half of this option fails the same `todo.md:407` strict reading as Option 1 (Codex CORRECTION above). launchd itself remains coherent as *optional hardening layered on Option 2* (supervisor + fail-closed launcher, no settings env — Codex's independent recommendation), at the cost of everything in this paragraph; Codex's own cross-check concedes the moving-parts penalty is real.

## Decision Heuristics

For reference, these are the priorities for choosing an approach:
1. Match existing codebase patterns over introducing novel approaches
2. Simpler is better — fewer files, fewer abstractions, fewer moving parts
3. Reversible over optimal — prefer approaches that can be easily changed later

## What We're NOT Doing

- No user-facing Vertex/ADC install path — machinery ships inert with a hidden relay toggle; no docs in user-facing text.
- No automation of the machine layer — VibeProxy app install, server start, provider OAuth logins are check-and-guide only.
- No changes to the `codex exec` trio (`/codex-research`, `/codex-audit`, `/design` integration) — AC8.
- No task 23 work (workflow model routing) — agent-type names are frozen for it, nothing more.
- No Windows/Linux support — VibeProxy is macOS-only; the whole lane is macOS-bound.
- No general fix for issue #3 (fresh-install permission gaps) — this task only writes the rules its own doctor/boot path needs.
- Not shipping `test-codex-leg.mjs` / `test-concurrent.mjs` — they stay dev-only; the doctor's preflight uses curl + in-session agent spawns.
- No relay feature work beyond the canonical two-variant merge (hardening already exists and ships as-is).
- No launchd by default — and therefore no `uncaughtException` fail-fast flip (coupled only to supervision, `todo.md:406`; keep-serving stays correct unsupervised). launchd is named as the future hardening path if mid-session relay crashes prove real in practice.
- No SessionStart-hook boot lane — rejected on the `todo.md:407` strict reading (timing itself verified fine; evidence preserved under Option 1 for any future revival).

## Decision

**Chosen approach:** Shared spine + Option 2 — installed fail-closed launcher (`~/.claude/native-agents/bin/claude-native`), no env in any settings file, no launchd by default.

**Developer sign-off:** confirmed 2026-06-11 (opt-in launcher explicitly preferred over the zero-command settings-env alternative).

**Rationale:** The decisive evidence is constitutional: `todo.md:407`'s parenthetical "(or launches plain `claude`)" plus "must never lock anyone out of plain sessions" rules out settings-file env under a strict reading — surfaced by Codex's cross-check (CORRECTION, verified against the spec text) and corroborated by the verified design's own framing (`codex-native-agents.md:117`). That eliminates Options 1 and 3 as written and resolves the task's own open question at `todo.md:442` in the wrapper direction. Among what remains, the launcher wins on all three heuristics: **patterns** — it productizes the dev machine's proven `claude-codex()` wrapper shape (codex doc §5A) and installs via the Step 3B copy/diff precedent; **simplicity** — no hooks, no daemon, no settings env, one new executable file; **reversibility** — delete the launcher and agent files and the machine is stock. The fail-closed gate converts the headline lockout risk from "mitigated" to "structurally impossible" (env is never set against an unverified relay). Codex's cross-check materially shaped this decision twice: the 407 correction, and the installed-launcher refinement that fixed Option 2's shell-rc weakness. I diverge from Codex's Phase 3 on one point: launchd is dropped from the default install — the launcher alone already guarantees relay-up at every relayed-session start, mid-session crash recovery is the only marginal benefit, and Codex's own RISK assessment concedes the moving-parts penalty; per the simplicity and reversibility heuristics, supervision ships later if real-world crashes justify it. Accepted residuals: per-session opt-in habit (`claude-native` instead of `claude` — doctor step 0 and agent-description notes make forgetting diagnosable), unsupervised mid-session crashes (CLI retry-with-backoff is the cushion), and the Gemini OAuth lane's by-design untested status (the doctor probe is the per-user verification).
