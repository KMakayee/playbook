# Design: Task 23 — Multi-model workflow routing (CLAUDE.md Workflow section + recursion-guard upgrade + /forge rewrite)

## Context

Task 22 shipped GPT-5.5 and Gemini 3.5 Flash as native Claude Code subagent types (`codex`, `codex-xhigh`, `gemini-flash`) behind a local relay. Task 23 adds the routing layer on top: (1) a new **Workflow** section in CLAUDE.md governing model allocation for ALL agent dispatch (Workflow-tool scripts and plain Agent-tool spawns); (2) replacement of the absolute recursion guard with a developer-supplied orchestrator-grant version (non-transitive, adaptive-investigation-only); (3) a rewrite of `/forge` into a piece-agnostic, dispatch-led orchestration lane on the new allocation.

Most of the *allocation* is developer-decided and not open for design: role defaults (Codex = coding, Opus = audit/synthesis, Gemini Flash = small/volume/fetching, Sonnet = trivial harness-dependent fan-out, session model = orchestrator only), the artifact authoring chain (Codex ideation → Opus authors → orchestrator reviews), Codex-builds-all-code, piece-agnostic forge, and the replacement guard text. What design decides is **presentation** (section shape, placement, cue wording, fallback verbosity), **forge orchestration mechanics** (per-piece dispatch composition, what survives from the current lane), and **four reconciliations** (guard ↔ forge caps/leaf-write, routing ↔ Sub-Agent Use, gates ↔ exec, fallback ↔ no-env-sniffing).

Hard constraints (from the task spec, all verified in research): cue-based soft guidance with a freedom clause, never spawn the session model as a sub-agent, reviewer ≠ author with Gemini as third-family escalation, fast-fail-as-detector fallback (no env sniffing; per-lane, not binary), correct dispatch mechanics (`model:` enum `sonnet|opus|haiku|fable` for Claude tiers; `agentType` for Codex/Gemini), trio stays on `codex exec`, both CLAUDE.md edits mirrored into `.claude/templates/playbook-sections.md`, no other skill touched (AC6), non-code pieces first-class in forge with per-type Verify.

**Research:** `tasks/research-codebase.md` (16 axes, 9 couplings, 6 synthesis findings)

## Options Considered

### Option A — Route-only (zero task-22 template edits)

**Axis-choice combination:** spine + **7(a)** Bash-writes, **8(a)** standing-request sentence, **16(b+c)** docs sweep without the playbook-setup sentence.

How it works: the Workflow section, guard replacement, mirror, and forge rewrite land as pure routing-layer edits. The three agent templates ship unchanged (`tools: Read, Glob, Grep, Bash`; descriptions still say "use only when explicitly requested"). Builders write files via Bash (heredocs, `python - <<EOF`, patch application) — GPT-5.5's native Codex-CLI mode is shell-mediated patching, so this is close to its home turf. The Workflow section adds one sentence declaring that the routing table **constitutes the standing explicit request** the agent descriptions ask for — reconciling the description gate without touching it. Docs: README `/forge` row + workflow blurb ("two-agent handshake" → three-family allocation) + native-agents cross-ref + `quickref.md:28`.

Good: smallest diff; no task-22 file touches; no propagation chain (no re-install nudge, no project/user copy drift); the conservative description gate keeps protecting **non-playbook projects** — user-level `~/.claude/agents/` copies leak into every project on the machine, and "use only when explicitly requested" is the only thing stopping codex auto-selection in repos with no Workflow section.

Not good: Bash-only writes forgo the Edit tool's exact-match safety and the harness's file-state tracking/diff display — builder writes are opaque commands in the transcript; in consumer permission setups, arbitrary Bash writes may hit prompts that Edit/Write would not (a stall inside a background Workflow lane is hard to see); the standing-request sentence asks readers to hold a two-document reconciliation in mind (description says "only when explicitly requested," CLAUDE.md says "this is the request"). **Codex cross-check resolved the blocking question against this option with in-repo evidence:** the permission-prompt failure mode is not hypothetical — the playbook already tracks it as standing issues (fresh installs hit Bash permission gaps that hard-stop workflows one denial at a time, `tasks/new-issues.md:60`; headless children hang on permission prompts, which is the only reason `--dangerously-skip-permissions` exists in the implement skills, `tasks/new-issues.md:205-223`), the native-agents install allowlists only doctor/probe Bash commands — nothing write-capable (`native-agents/SKILL.md:90-97`), and the trio skills' tmp-file compose pattern exists precisely because shell quoting is a recognized hazard in this repo (`codex-review/SKILL.md:35`).

### Option B — Builder-aligned (templates ride the existing propagation chain)

**Axis-choice combination:** spine + **7(b, uniform per developer decision 2026-06-11)** extend `tools:` with `Edit, Write, LSP, WebFetch, WebSearch` on all three templates, **8(b)-lite** minimal description amendment on all three, **16(c)** docs sweep without the playbook-setup sentence.

*(History: as originally drafted this option extended all three templates and included a `playbook-setup/SKILL.md:29` edit — both narrowed by Codex's cross-check: the all-three scope was internally inconsistent with the open question hedging on gemini, and the setup enumeration sentence is not materially stale — it already declines to enumerate Sub-Agent Behaviors and Quality Standards. The developer subsequently (2026-06-11) restored and extended the all-three scope deliberately — uniform tools across all three agents, resolving the inconsistency in the opposite direction — while the playbook-setup trim stands.)*

How it works: everything in Option A's routing layer, plus a one-line `tools:` change on **all three** templates — `Read, Glob, Grep, Bash` → `Read, Edit, Write, Glob, Grep, LSP, WebFetch, WebSearch, Bash` (uniform allocation per developer decision, 2026-06-11) — so code writes go through the harness's canonical path, workers get precise code navigation (LSP is harness-side: works regardless of upstream model, and degrades to a contained error where no language server is configured), and the fetch lane is first-class. A minimal description amendment on the same three files **keeps the explicit-request gate** while acknowledging routing: append "…or routed by a project's CLAUDE.md Workflow/routing section" to the existing "use only when explicitly requested" clause. The relayed-session requirement and contained-error warning stay verbatim (the doctor relies on that framing). With tools uniform across the three agents, **role discipline is carried entirely by the routing table** — the table's recognition cues, not tool availability, are what allocate work; the uniform grant also makes gemini's "small/minor tasks" role coherent for small mechanical edits. **Caveat carried into the plan:** web-tool behavior through the relay is unverified for non-Anthropic upstreams — WebFetch is harness-side (fetch + summarize via the session's API) and should work; WebSearch may be Anthropic-server-side and could fail on the relayed lanes. Both ship anyway: a failed web call inside a relay-served worker is a contained per-lane failure, and the Workflow section's fallback cue already routes web work to Claude-harness agents when a lane can't do it. A doctor probe for the web tools is a **deferred follow-up**, not in scope (keeps "no doctor changes" intact). The CLAUDE.md standing-request sentence (carried over from Option A, needed anyway for the drift window on older installed copies) still ships. The change rides the existing propagation chain (Coupling 2): templates → `/native-agents install` re-copy → project + user-level copies, with `/playbook-update`'s re-install nudge (`playbook-update/SKILL.md:133`) as the consumer channel; the doctor/drift-checklist text doesn't reference `tools:` lines, so no contradiction there. Docs: README `/forge` row + workflow blurb + native-agents cross-ref + `quickref.md:28`; `playbook-setup/SKILL.md` untouched.

Good: builders use the harness's canonical write path — exact-match Edit safety, file-state tracking, user-visible diffs, predictable permissioning (the in-repo evidence under Option A's "not good" cuts in this option's favor: Edit/Write sidesteps the documented Bash permission-gap failure mode entirely); "Codex builds all code" (the load-bearing developer decision) gets the same write mechanics every other writer in the playbook uses; the description amendment resolves the routing-by-default tension at its source for the agents that routing makes default, while preserving the gate for unrouted projects; the uniform tools line means no per-agent reasoning about capability gaps — any lane can take any work the table routes to it.

Not good: touches three task-22-shipped files; gemini gains a write surface its routing role only thinly uses (accepted by developer decision — simplicity over minimal-surface discipline); installed copies (project + user level) are stale until a re-install — drift window where the table says "builders write" but deployed agents lack Edit/Write (mitigated twice: Bash-writes still work in the interim so degradation is to Option A's behavior, not breakage; and the standing-request sentence keeps routing legible against the old descriptions); the propagation checklist must be exercised (install re-run on this machine, re-install nudge for consumers).

### Option C — Patch-return builders (leaves stay read-only)

**Axis-choice combination:** spine + **7(c)** builders return diffs/patches as structured output; orchestrator applies, **8(a)** standing-request sentence, **16(b+c)** docs sweep as in A.

How it works: templates unchanged; builders produce unified diffs (Workflow `schema` forces structured output), and the orchestrator (or an Opus applier sub-agent) applies them in the main loop via Edit/Write. Parallel lanes stop needing `isolation: 'worktree'` since leaves never mutate.

Good: preserves task 22's leaf-reader design intent most faithfully; every write goes through harness Edit safety under the orchestrator's eyes; zero template edits; no worktree merge-back problem (research Risk 6 dissolves).

Not good: re-centralizes the writing the task explicitly delegates ("the orchestrator frames, coordinates seams, and reviews rather than writing the code itself" — applying every patch is writing); full diffs flow through the main-loop context, colliding head-on with the compaction rules' 30% budget on any wide piece (the exact scenario forge exists for); doubles latency per seam (build, then apply); patch application failures (context drift between generation and apply) become an orchestrator burden the other options don't have. The structural contradiction with the developer decision is disqualifying unless Codex's cross-check surfaces something the other two can't handle.

## Decision Heuristics

For reference, these are the priorities for choosing an approach:
1. Match existing codebase patterns over introducing novel approaches
2. Simpler is better — fewer files, fewer abstractions, fewer moving parts
3. Reversible over optimal — prefer approaches that can be easily changed later

## Decision

**Chosen approach:** Option B — builder-aligned, with the uniform-tools developer decision (2026-06-11): the 13-axis shared spine + all three templates (`codex.md`, `codex-xhigh.md`, `gemini-flash.md`) gain `Edit, Write, LSP, WebFetch, WebSearch` + the routing-aware description amendment on all three + docs scope 16(c) (README row/blurb/cross-ref + quickref; no playbook-setup edit). Final tools line, identical across all three: `Read, Edit, Write, Glob, Grep, LSP, WebFetch, WebSearch, Bash`.

**Rationale:** Codex independently converged on the identical 13-axis spine and initially chose Option A's fork choices on simplicity grounds — then reversed on Axis 7 after resolving the blocking question with in-repo evidence rather than speculation: the Bash permission-gap failure mode is already a documented standing issue in this repo (fresh-install denials hard-stop workflows, `tasks/new-issues.md:60`; headless children hang on prompts, `tasks/new-issues.md:205-223`), and the native-agents install allowlists nothing write-capable (`native-agents/SKILL.md:90-97`). On the decision heuristics: Edit/Write is the harness's **existing pattern** for file mutation (exact-match safety, file-state tracking, visible diffs) versus the novel Bash-heredoc write path, so heuristic 1 favors B once the simplicity gap narrows to two one-line frontmatter edits; the narrowed scope keeps the diff small (heuristic 2) and the change reversible (heuristic 3) — the interim drift window degrades to Option A's still-working Bash-writes, not breakage. Option C was disqualified by its structural contradiction with the developer decision (orchestrator applying every patch is the orchestrator writing) plus the main-loop context-budget collision. Codex's cross-check materially shifted the fork choice (A→B on write mechanics) and trimmed B's scope twice (gemini untouched; playbook-setup untouched) — convergence on everything else.

**Amendments (2026-06-11, developer):** after the cross-check, the developer flattened the per-role tool split into a uniform grant — all three templates get `Edit, Write, LSP, WebFetch, WebSearch` — trading minimal-surface discipline for simplicity and forward-compatibility (no per-agent reasoning about capability gaps; gemini's small-mechanical-edits role becomes coherent). This consciously overrides the cross-check's gemini-untouched trim; the playbook-setup trim stands. With tools uniform, role allocation is carried entirely by the Workflow section's routing table and recognition cues — tool availability no longer differentiates the lanes. The unverified-WebSearch-through-relay caveat is logged as a non-blocking open question with a deferred doctor-probe follow-up.
