## Tasks

- [x] 1. **Make `/checkpoint` reliable for suspend and resume** — Rework checkpoint to support two modes: (a) a waypoint inside a QRSPI cycle that automatically captures the current phase and step/batch so a later session resumes at the exact point, and (b) a standalone snapshot usable outside QRSPI for quick saves. State capture must happen automatically — no relying on the user or agent to narrate where they are. Checkpoints commit to git on creation so they double as backups. On resume, the artifact is consumed and deleted after rehydration so stale checkpoints don't accumulate. Motivation: the command today misses workflow position, doesn't work cleanly outside QRSPI, and leaves old artifacts lying around. This also unlocks reliable batching of large plans across multiple implementation sessions — `/checkpoint` becomes the vehicle for suspending between batches. **Before shipping to regular use, run an end-to-end test: create a checkpoint mid-batch, exit the session, resume in a fresh session, and verify that state rehydrates cleanly and the artifact is consumed. Breakage here loses in-flight work silently, so dogfood before promoting to production.**

- [x] 2. **Ship a generalized Codex review entry point** — Add a simple `/codex-review <target>` for ad-hoc second-opinion reviews. Consolidate the unbiased-review principles already encoded across the existing Codex-driven review prompts (research, plan, code review) into one generalized prompt so any target — a file, a diff, an artifact — gets the same framing. Purpose: replace ad-hoc Codex invocations that drift from the principles established elsewhere in the playbook. Keep it simple — no task-lifecycle integration, just a shortcut.

- [ ] 8. **Add `/catchup` for parallel-worktree catch-up to the default branch** — When multiple worktrees branch from the same default branch (`main`/`dev`) and merge sequentially, the second-and-later branches predictably go stale once a sibling lands. Today this requires manual catch-up: merge default into the feature branch, resolve conflicts by hand, run host-project validation (build/test), then push. Add a standalone `/catchup` command that handles the flow uniformly — drift detection → merge default in → surface conflicts with guidance → run validation hook → ready to push. Should pair with a small staleness gate in `/push-pr` and `/push-pr-light`: if the feature branch is behind the default, refuse and point the developer at `/catchup` rather than embedding catch-up logic into the push commands themselves. **Useful context to fold into research:**
  - **Merge over rebase as default.** No force-push needed; the eventual squash on PR collapses the merge commit so it doesn't pollute the default branch's history. Rebase is opt-in for branches not yet pushed.
  - **Conflict-resolution failure mode.** The dominant way conflicts go wrong is accidentally dropping one side's line during manual resolution. For *independent* additions on both sides (e.g., two new `package.json` scripts at the same anchor), keep BOTH sides — pick one only when the values are mutually exclusive. The conflict prompt should surface this guidance explicitly.
  - **Reversibility safety net.** Standard git escape hatches exist both pre-commit (`git merge --abort`) and post-commit-pre-push (`git reset --hard ORIG_HEAD`). The conflict prompt should surface these so devs know the bailout path is cheap.
  - **Worst-case recovery.** Even if a line gets accidentally dropped during resolution, the original feature-branch commits are untouched (the merge is additive — original SHAs survive). Recovery via `git show <feature-sha>:<path>` is always possible.
  - **Validation hook is host-project specific.** Build/test commands should come from project config (`CLAUDE.md` or similar), not be hardcoded. Skip the validation step if no config exists (the playbook itself has nothing to validate, but most consuming projects do).
  - **Default-branch detection.** Should auto-detect rather than hardcode `main` (some projects use `dev` or other names). Fall back to asking the developer if detection fails.
  - **Slash-command call constraint.** Slash commands cannot programmatically invoke other slash commands (same constraint that prevents `/checkpoint` from auto-running `/compact`). `/push-pr`'s staleness gate must inspect git state and *tell* the developer to run `/catchup`, not invoke it directly.
  - **Idempotency.** Running `/catchup` when already up to date is a no-op — one-line "already up to date" report, then return cleanly.
  - **Closes Issue #1.** Issue #1 in `tasks/issues.md` was originally scoped to bundle catch-up + artifact cleanup + squash default into `/push-pr`. Task 8 absorbs catch-up and artifact cleanup; Issue #1 has been re-scoped to just the squash default. Task 8 should close Issue #1 out by handling the squash-default change in `/push-pr` and `/push-pr-light` alongside the staleness-gate edit (both are small `/push-pr` changes — co-locate).
