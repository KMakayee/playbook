- [x] 8. **Add `/catchup` for parallel-worktree catch-up to the default branch** — When multiple worktrees branch from the same default branch (`main`/`dev`) and merge sequentially, the second-and-later branches predictably go stale once a sibling lands. Today this requires manual catch-up: merge default into the feature branch, resolve conflicts by hand, run host-project validation (build/test), then push. Add a standalone `/catchup` command that handles the flow uniformly — drift detection → merge default in → surface conflicts with guidance → run validation hook → ready to push. Should pair with a small staleness gate in `/push-pr` and `/push-pr-light`: if the feature branch is behind the default, refuse and point the developer at `/catchup` rather than embedding catch-up logic into the push commands themselves. **Useful context to fold into research:**
  - **Merge over rebase as default.** No force-push needed; the eventual squash on PR collapses the merge commit so it doesn't pollute the default branch's history. Rebase is opt-in for branches not yet pushed.
  - **Conflict-resolution failure mode.** The dominant way conflicts go wrong is accidentally dropping one side's line during manual resolution. For *independent* additions on both sides (e.g., two new `package.json` scripts at the same anchor), keep BOTH sides — pick one only when the values are mutually exclusive. The conflict prompt should surface this guidance explicitly.
  - **Reversibility safety net.** Standard git escape hatches exist both pre-commit (`git merge --abort`) and post-commit-pre-push (`git reset --hard ORIG_HEAD`). The conflict prompt should surface these so devs know the bailout path is cheap.
  - **Worst-case recovery.** Even if a line gets accidentally dropped during resolution, the original feature-branch commits are untouched (the merge is additive — original SHAs survive). Recovery via `git show <feature-sha>:<path>` is always possible.
  - **Validation hook is host-project specific.** Build/test commands should come from project config (`CLAUDE.md` or similar), not be hardcoded. Skip the validation step if no config exists (the playbook itself has nothing to validate, but most consuming projects do).
  - **Default-branch detection.** Should auto-detect rather than hardcode `main` (some projects use `dev` or other names). Fall back to asking the developer if detection fails.
  - **Slash-command call constraint.** Slash commands cannot programmatically invoke other slash commands (same constraint that prevents `/checkpoint` from auto-running `/compact`). `/push-pr`'s staleness gate must inspect git state and *tell* the developer to run `/catchup`, not invoke it directly.
  - **Idempotency.** Running `/catchup` when already up to date is a no-op — one-line "already up to date" report, then return cleanly.
  - **Closes Issue #1.** Issue #1 in `tasks/issues.md` was originally scoped to bundle catch-up + artifact cleanup + squash default into `/push-pr`. Task 8 absorbs catch-up and artifact cleanup; Issue #1 has been re-scoped to just the squash default. Task 8 should close Issue #1 out by handling the squash-default change in `/push-pr` and `/push-pr-light` alongside the staleness-gate edit (both are small `/push-pr` changes — co-locate).

- [x] 9. **Reshape `/codex-review` triage output to recommend-only** — Today Step 6b of `.claude/commands/codex-review.md` presents *all* findings with apply / judgment call / noise labels, including narrating why noise items are noise. Reshape to recommend-only by default: list **apply** items as actual recommendations, **judgment call** items as a "needs your input" subsection, and collapse **noise** to a count with a `show all` opt-in for the full list. Motivation: triage is the dominant Claude-side cost in `/codex-review` and the current full-list view spends tokens narrating items the developer will skip anyway — moving to recommend-only cuts the presentation surface to the items Claude actually wants to defend. - [ ] 10. **Migrate long-running Codex and `claude -p` calls to background-by-default** — Today most QRSPI commands invoke Codex foreground with a 10-minute Bash timeout (`research-codebase`, `design`, `create-plan`, `issue-research`, `issue-plan`, plus the structural-mismatch paths in `implement` and `issue-implement`); only `/implement` Step 6/8, `/issue-implement` Step 6/8, and `/auto-issues` Phases 1-3 use `run_in_background: true`. The foreground 10-minute cap is a silent failure mode — if a Codex run ever exceeds 10:00, the harness kills the process and Claude proceeds on truncated output with no error. Migrate every long-running Codex and `claude -p` invocation to backgrounded mode so the cap stops mattering, and bake `</dev/null` redirection into every backgrounded site to close the stdin-pipe hang surfaced by Issue #2. Add a CLAUDE.md rule that any backgrounded subprocess must redirect stdin to `/dev/null` so future call sites stay safe by default. **Useful context to fold into research:**
  - **Migration targets.** Foreground Codex calls in `.claude/commands/research-codebase.md`, `design.md` (three sites — primary research, blocking-question deep-dive, pattern-research), `create-plan.md`, `issue-research.md`, `issue-plan.md`, and the structural-mismatch Codex calls in `implement.md` and `issue-implement.md`. Foreground `claude -p` calls in `auto-issues.md` Phase 4 (Update) and Phase 5 (Commit & Push). Line numbers will drift during migration — grep `'10-minute timeout'` and `'Timeout: 600000ms'` to locate every site.
  - **Leave foreground (opt-in only).** `/codex-review` is a one-shot quick review with no persistent artifact — backgrounding makes small reviews feel heavier. Keep it foreground by default; allow an explicit opt-in flag if the developer wants to background a long review.
  - **Verify auto-resume empirically before declaring it uniform.** The existing background pattern in `/implement` (Step 6 → Step 7 handoff) relies on the harness auto-notifying Claude when the background task completes — that's an external harness assumption, not repo-proven behavior; the output check script (`.claude/scripts/codex-output-check.sh`) only validates file existence and line count, not active waiting. Before migrating wholesale, run a few `/research-codebase`, `/design`, `/create-plan` invocations in background mode and confirm Claude resumes the next step without user nudging. If auto-resume is flaky on some path, fall back to an explicit completion-check script invoked from the next step.
  - **`/research-codebase` is the weakest UX win.** Codex is the first substantive step; Claude has nothing concurrent to do during the wait. The benefit there is purely "no 10-min cap," not parallelism — set expectations accordingly.
  - **Out of scope.** Non-CLI long-running operations like `git merge` in `/catchup` and `git fetch --unshallow` in `/playbook-update` have different semantics (interactive conflict handling, no stdin-probe behavior) — foreground is correct for those, leave alone.
  - **Issue #2 stays separate.** Issue #2 ships first as the operational hotfix (`</dev/null` on the 7 existing background sites + regression-guard grep). Task 10's migration absorbs those sites into the broader pattern; its acceptance criteria should validate they still carry `</dev/null` after the broader migration lands. Do not delete Issue #2 in favor of Task 10 — the hang is High-priority operational pain that needs to ship before the broader redesign.
  - **Sequencing.** Land before Task 7 (skill port) so the port reflects the new pattern. Task 5 also touches `research-codebase` and CLAUDE.md; either land Task 10 before Task 5 (treat the migration as foundational since the 10-min cap is real operational pain) or after Task 5 (avoid merge churn on overlapping files).
