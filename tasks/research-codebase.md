### Upfront spec

**Intent.** Add a model-routing layer for delegated work, now that Codex and Gemini run as native subagent types (task 22). Three pieces. (1) A new **Workflow** section in CLAUDE.md governing ALL agent dispatch — Workflow-tool scripts AND plain Agent-tool spawns: when work is delegated, the orchestrator allocates the model by role. Role defaults: **Codex = coding** — essentially all of it, including small code changes; **Opus = auditing Codex output, synthesis, review**; **Gemini Flash = small/minor tasks, repetitive and high-volume/high-frequency work, and fetching**; **Sonnet = trivial fan-out work that specifically relies on the Claude harness** (reliability, harness tooling, longer context — volume alone goes to Gemini); **the session model (Fable) = orchestrator only, never spawned as a sub-agent**. The role defaults compose into a standing **artifact authoring chain** for non-code deliverables (specs, docs, plans, strategy): Codex = upstream generative work (knowledge dump, ideation, strategy, mockups); Opus = authors the deliverable from that raw material; the orchestrator = reviews in the main loop — stated here at the routing level so every dispatching workflow inherits it, not just `/forge`. Written as cue-based soft guidance — guardrails plus stated-reason freedom, not a rigid matrix (workflows hit millions of scenarios; the section steers defaults, it doesn't forbid judgment). (2) Replace the recursion guard with the orchestrator-grant version (text below). (3) Rewrite `/forge` — the only existing workflow-touching skill — into a dispatch-led orchestration lane on the new allocation. Three developer decisions (2026-06-11): **Codex builds all code** — even the main build pass dispatches to Codex workers; the orchestrator frames, coordinates seams, and reviews rather than writing the code itself. **Forge is piece-agnostic and task-shaped**: the piece may be a non-code deliverable (spec, design doc, plan) just as well as code, and the orchestration is composed per piece (sequential seams, parallel Workflow lanes, a single worker — whatever the piece's shape calls for) rather than one fixed linear pass. **Artifact pieces follow the authoring chain** defined at the routing level in (1) — Codex ideation/knowledge-dump → Opus authors → orchestrator reviews; reviewer≠author holds at every link, and the Codex gates still run on the Opus-authored artifact. Forge applies the chain; it does not define it. Invocation stays `/forge <piece>` with no implied pairing or sequence: forge whatever the piece is — specs only for a while, code from a spec forged weeks ago, a doc, a plan, something else entirely. Chained runs (`/forge` the spec, later `/forge` the code from it) are one common pattern where run 1's output becomes run 2's source — an emergent usage, not a prescribed two-step, and never one internal authorship pipeline.

**Constraints.**
- Routing is guidance with guardrails: recognition cues + default allocations + an explicit freedom clause (deviate with a one-line stated reason). Per established feedback: concrete cues over "best judgment" — every role gets recognition cues, not adjectives.
- The session model is never spawned as a sub-agent. Orchestration, spine decisions, and user interaction stay in the main loop.
- Acknowledge the Opus/Codex overlap (both review and synthesize well) with a cross-model cue: **reviewer ≠ author** — Codex-written code is audited by Opus/Claude; Claude-written artifacts get Codex review (the existing playbook handshake). Don't pretend the roles are disjoint. Escalation layer: on high-stakes items, add a **Gemini verification pass** as a third perspective family-independent of both Codex/GPT and Claude — an additional vote on top of the handshake, never a replacement for the primary reviewer.
- **Graceful degradation, no env sniffing:** in a non-relayed session, `agentType: codex` / `gemini-flash` fails fast with a distinctive error (~440ms — codex doc §7b); that fast-fail IS the installed-or-not detector. Fallback routing is Claude-only (Opus takes coding, Sonnet/Haiku take small tasks), and `/forge` falls back to its current orchestrator-builds-in-main-loop shape. The playbook must remain fully usable without task 22's install.
- Mechanics stated correctly: the Workflow tool's `model:` opt is a closed enum (`sonnet|opus|haiku|fable`) — Codex/Gemini route via `agentType`; Claude tiers via `model`. Cite the agent-type names task 22 ships.
- The trio (`/codex-research`, `/codex-review`, `/codex-audit`) stays on `codex exec` — explicitly out of scope (codex doc §7b remains design-only).
- Recursion-guard replacement is the developer-supplied orchestrator-grant text (below) — adapt wording minimally; the grant is non-transitive (max depth: parent → orchestrator → leaf); granted only for adaptive investigation; plannable fan-out returns a work-list as structured output instead.
- `/forge`'s leaf-write override, sub-agent caps, and "every spawn pins model explicitly" rule must be reconciled with both the new guard and the new builder routing — not deleted, re-derived.
- Non-code pieces are first-class, not incidental: Frame classifies **deliverable type** alongside load-bearing/conform-only, and Verify gets a defined per-type form — tests/typecheck for code; fidelity audit against upstream sources + internal-consistency pass for artifacts (specs, docs, plans). The type list is open: a piece that is neither gets a Frame-declared verify plan rather than a forced fit. Today an artifact piece silently loses its verify loop (`forge/SKILL.md:47` hard-codes tests/typecheck) — that silent degradation is the bug this constraint fixes.
- CLAUDE.md edits (Workflow section AND new guard) are mirrored into `.claude/templates/playbook-sections.md` (`playbook-setup/SKILL.md:31` appends it on existing-project setup; a CLAUDE.md-only edit is lost on fresh installs).
- Depends on task 22 (the agent types and relay this section routes to must be installable first).

**Replacement recursion-guard text** (developer-supplied; adapt minimally):
> **Recursion guard:** Sub-agents are leaf tasks (read, search, report) and MUST NOT spawn sub-agents unless their spawn prompt explicitly grants the orchestrator role. The grant is non-transitive: an orchestrator spawns leaves only (max depth: parent → orchestrator → leaf).
> Grant the role only for adaptive investigation — when each next spawn depends on the previous result and the lead's accumulated context doesn't serialize well (deep debugging, unfamiliar-code archaeology). If the fan-out is plannable up front, don't grant: have the lead return a work-list as structured output and spawn its items from the parent or workflow script.

## Summary

The task is three deliverables but most of the *allocation* is already developer-decided (role defaults, authoring chain, Codex-builds-all-code, piece-agnostic forge). What design actually decides is: **presentation** (cue wording, table shape, section placement, fallback verbosity), **forge orchestration mechanics** (how dispatch is composed per piece, what survives from the current lane), and **four reconciliations** the spec demands (guard ↔ forge caps/leaf-write, routing ↔ Sub-Agent Use, gates ↔ exec, fallback ↔ no-env-sniffing).

Codex's sweep held up under spot-checking — every sampled file:line claim verified (relay routing, launcher fail-closed exec, kept-doc mechanics, same-family-review cost note, managed-list entries). Its 13 axes are genuinely axis-shaped; synthesis added three more it missed, plus couplings.

The load-bearing synthesis findings:

1. **The builder agents have no Edit/Write tools.** All three shipped agent templates carry `tools: Read, Glob, Grep, Bash` — chosen by task 22 for the *leaf-reader* role. Task 23 repurposes `codex` as the universal **code builder**, which means workers write files via Bash only (heredocs, `python - <<EOF`, patches) or the templates change. This is a real design axis (Axis 7), not a footnote: it touches task-22-shipped files, the model-ID-drift propagation checklist, and user-level copies. Mitigating fact: GPT-5.5 is trained on shell-mediated patching (Codex CLI's native `apply_patch` flows through shell), so Bash-writes are closer to its native mode than to a workaround.
2. **The shipped agent descriptions say "Use only when explicitly requested."** All three templates' `description:` fields end with "Use only when explicitly requested for codex-powered workflow agents" — written before routing-by-default existed. Agent descriptions steer the harness's auto-selection of `subagent_type`; a CLAUDE.md section that makes Codex the *default* coder sits in direct tension with descriptions that gate on explicit request (Axis 8).
3. **The never-spawn-the-session-model rule needs scope wording.** The Workflow `model:` enum *includes* `fable`, and the Workflow tool's own guidance says "default to omitting `model` — the agent inherits the main-loop model" — so under Fable, the harness's default behavior produces exactly the Fable swarm the rule forbids. The explicit-pin rule is therefore the *enforcement mechanism*, not a style preference. But in a non-Fable session (Opus-led stock session), a literal "never spawn the session model" collides with the fallback table ("Opus takes coding") — the rule needs deliberate scoping (Axis 6).
4. **Distribution is a non-problem — verified.** `/playbook-update` Category B merges everything from the `# RDPI Workflow Rules` heading onward as the playbook-owned bottom half (`playbook-update/SKILL.md:137-156`), so the new Workflow section propagates to consumers wherever it lands within that region; `playbook-sections.md` is already managed (`playbook-update/SKILL.md:22`). No new files are expected → no managed-list additions.
5. **The forge rewrite is a re-derivation, not a green-field.** The contradictions with task 23 are localized: the routing hierarchy (`forge/SKILL.md:57-61`), the code-only Verify (`:47`), and the "model-led single pass" framing (`:3`, `:11`). The gate cycle, triage buckets, continuation-prompt discipline, cleanup contracts, and design-confirm pause all survive structurally.
6. **Live-session verification nuances** (this research session runs relayed in the task-23 worktree): `.claude/agents/` is gitignored (`.gitignore:6`), so worktree-launched sessions resolve the agent types from **user-level** `~/.claude/agents/` copies only — a relayed session in a fresh worktree without the global install fast-fails despite the repo being "installed." And degradation is **per-lane**, not binary: the gemini provider is optional in VibeProxy, so `codex` can pass while `gemini-flash` fast-fails on the same machine — the fallback rule must handle a partial lane, not just installed/not-installed.

### 3. Harness ground truth — Agent tool and Workflow tool APIs (verified from live session schemas)

These are facts about the runtime the routing section instructs, not about repo files; verified directly from this session's tool schemas (no public URL — see External Research).

- **Agent tool:** `subagent_type` selects the agent; custom types resolve from `.claude/agents/` (project) and `~/.claude/agents/` (user), project precedence, loaded at session start. `model` override accepts exactly `sonnet | opus | haiku | fable`. `isolation: 'worktree'` exists. The session's agent listing carries each custom agent's `description:` — this is what steers auto-selection, which is why the templates' "use only when explicitly requested" wording matters (Finding 6c).
- **Workflow tool:** `agent(prompt, opts)` accepts `opts.model` (same closed enum), `opts.agentType` (custom subagent type by name — "resolved from the same registry as the Agent tool"), `opts.isolation: 'worktree'` ("EXPENSIVE … use ONLY when agents mutate files in parallel"), `opts.schema` (structured output — kills temp-file plumbing for findings). The tool's own guidance on `model`: "**Default to omitting it** — the agent inherits the main-loop model … Only set it when you're highly confident." Under a Fable session, the harness default therefore produces Fable workers — the explicit-pin rule is the only thing standing between a routing table and an accidental Fable swarm. `/forge` already counter-instructs this deliberately (`forge/SKILL.md:58`) but says only `model`; the rule must now cover `agentType` too.
- **Workflow structural facts** that align with the new guard: workflows take no mid-run user input (grounds AC4's "design-confirm pause stays main-loop"); `workflow()` nesting is one level only; concurrent `agent()` calls cap at min(16, cores−2) per workflow. The non-transitive grant (parent → orchestrator → leaf) matches the harness's natural depth limits rather than fighting them.

### 6. Gaps Codex's sweep missed (synthesis findings, all verified)

- **(a) Builder write mechanics.** `tools: Read, Glob, Grep, Bash` on all three agent templates — no Edit/Write. Task 22 designed them as leaf *readers/reporters*; task 23 makes `codex` the universal *builder*. Writing through Bash works (heredocs, `python`, patch application — and is GPT-5.5's native Codex-CLI mode) but forgoes the Edit tool's exact-match safety and the harness's file-state tracking; extending the templates' tools line is a one-line change per file but touches task-22-shipped artifacts with a 4-site propagation checklist (templates → project copies → user-level copies, via install re-run). Full axis below (Axis 7).
- **(b) Agent-description tension.** "Use only when explicitly requested for codex-powered workflow agents" (all three `description:` fields) was written for a world where spawning Codex was exceptional. Under the routing section, Codex is the *default* coder. Descriptions steer harness auto-selection; leaving them as-is undercuts the table in any session that consults descriptions over CLAUDE.md, while editing them touches shipped task-22 files (Axis 8).
- **(c) Session-model rule scope.** "The session model (Fable) = orchestrator only, never spawned." In a Fable session this is crisp. In an Opus-led stock session, the *session model is Opus* — and the Claude-only fallback table says "Opus takes coding." A literal reading forbids exactly what the fallback prescribes. The rule's evident purpose is "don't multiply the top-tier orchestrator" (cost + spine-ownership), not "never use opus workers" — but the section must say which it means (Axis 6).
- **(d) Per-lane degradation.** The gemini lane is optional and ships untested (task 22 decision); `codex` passing while `gemini-flash` fast-fails is a normal machine state, and `.claude/agents/`'s gitignore means worktree-launched sessions lack project copies entirely (user-level install is what saves them). The fast-fail detector and fallback routing should be worded per-agent-type, not as a binary installed/not-installed switch (folded into Axis 4).
- **(e) Empirical anchor for reviewer≠author.** `/implement-codex` documents "final-review independence is weakened … Codex review now reviews Codex-written code (same agent family — distinct sessions … but shared training distribution)" as a known experimental cost (`implement-codex/SKILL.md:13`). The routing section's reviewer≠author cue and the Gemini third-family escalation generalize a tension the repo has already named in writing.

## Architecture Analysis

- **Why the guard was absolute:** when it was written, sub-agents were exclusively *readers* feeding a synthesizing parent — there was no orchestration layer below the session. Leaf-only was the correct invariant, and it shaped skills (the implement lanes dropped sub-agents partly because the guard foreclosed Codex-from-inside-sub-agents). Task 22 added a second model family *as the workers*; task 23's grant mechanism is the minimal relaxation: depth stays bounded (matching the harness's own one-level workflow nesting), and the default remains leaf.
- **Why the agent templates look the way they do:** task 22's research framed them as "minimal no-subagents leaf prompt" workers with read/search tools — the doctor only needed self-report + one tool-use probe. The missing Edit/Write tools and the "explicitly requested" descriptions are *inherited design intent from the leaf-reader era*, not oversights — which is exactly why task 23 must consciously re-derive rather than silently inherit them.
- **Why forge is the only skill rewritten:** `/forge` is the only skill that *allocates models* (its Orchestration & routing section is a private routing table). The RDPI skills delegate *reading*, which the new table covers via inheritance (their spawns are bulk fan-out reads / targeted gap-fills — gemini/sonnet rows). The trio are Bash-mediated Codex calls, not agent dispatch, so the routing section governs them only via the pointer line (AC1).
- **The two-agent handshake generalizes, not breaks:** the playbook's identity is "Claude synthesizes, Codex cross-checks" (README:3, :100). Task 23 turns a 2-family handshake into a 3-family allocation with the same underlying rule — independence between author and reviewer. `implement-codex`'s documented same-family-review cost is the in-repo evidence the rule is load-bearing.
- **Economics as rationale, cues as behavior:** established posture — Codex calls cheap, Claude agent spend dear, Gemini Flash frontier-tier at a fraction of Claude prices. The intent text mandates this lands as one rationale line, with recognition cues carrying the actual behavior (per the concrete-cues-over-judgment feedback).

## Risk Analysis

- **The whole codex-builds-everything lane rests on undocumented passthrough.** A Claude Code release tightening frontmatter `model:` validation breaks every native lane at once; the routing section's fallback rule is then the *load-bearing* path, not an edge case. Wording should keep the fallback first-class (and the doctor is the detector — `native-agents/SKILL.md:205`).
- **Accidental Fable swarms via the inherit default.** Any workflow script that omits `model`/`agentType` spawns Fable workers silently. The section must make explicit-pin a hard rule with the failure mode named (forge already does this locally — `forge:58`; the rule is being promoted, and AC4 then *removes* forge's Fable-spawn allowance entirely).
- **Skill-local pins vs the new table.** RDPI skills pin `subagent_type: "Explore"` (e.g., `research-codebase/SKILL.md`) and similar — Claude-harness agents for work the table might route to gemini/sonnet. AC6 says inheritance-by-reading is the mechanism; the section needs a precedence sentence (skill-explicit instructions are a stated reason; the table fills the unspecified) or every RDPI run starts with a contradiction. Task OQ2 asks exactly this — it should be answered in the section text, not left ambient.
- **Per-lane partial degradation reads as flakiness.** Gemini optional/untested + worktree sessions lacking project agent copies (`.gitignore:6`) means one lane can fail while another passes on the same machine, same day. If the fallback is worded as a binary, users will read a gemini-lane fast-fail as "the install is broken" and abandon codex routing too.
- **Continuation prompts now carry routing state.** Forge's phase handoffs serialize spine decisions for a compacted window (`forge:50-53`); the rewrite adds dispatch composition (which seams went to which workers, what's mid-flight in a worktree lane) to what must survive the handoff. An under-specified continuation prompt strands a parallel lane.
- **Worktree merge-back is unowned.** `isolation: 'worktree'` gives parallel lanes isolated copies; nothing in forge or the Workflow tool merges them. If design leans on parallel lanes, the orchestrator's merge/apply step must be explicit in the skill, or lanes produce divergent worktrees the user must reconcile by hand.
- **Same-family review regression.** With Codex building all code, any lapse where Codex also reviews it reproduces the exact cost `implement-codex` documents (`implement-codex/SKILL.md:13`). The reviewer≠author cue has to be stated where dispatch happens (the table), not only in forge.
- **Task 13 soft collision.** Its bucket-vocabulary sweep edits forge's triage wording (`forge:45`); this task rewrites forge's routing/build/verify sections. Independent sections, but if 13 starts mid-flight, coordinate the forge file.
- **Doc drift if AC5 is read minimally.** `quickref.md:28` and `README.md:100` ("two-agent handshake") both go stale under the rewrite; AC5 names only the README row + frontmatter. Cheap to sweep now, confusing later (Axis 16).
