# Research: Task 7 — Port all playbook slash commands to skills

## Research Question

Port every playbook slash command (`.claude/commands/<name>.md`) into a Claude Code skill (`.claude/skills/<name>/SKILL.md`), relocate the supporting assets each command depends on so each skill is self-contained, and update every internal cross-reference so nothing breaks. Verify each converted skill by invoking `/<name>`. Additionally fold in the Deferred investigation: determine whether the RDPI-artifact restructure (static singleton `tasks/*.md` files → task-scoped directories) is cheap enough to land alongside Task 7, which would let skills ship with auto-invocation enabled rather than deferred.

### Upfront spec

- **Intent:** Convert each `.claude/commands/<name>.md` into `.claude/skills/<name>/SKILL.md` (body + YAML frontmatter), relocate supporting assets (`.claude/prompts/*.md`, `.claude/scripts/*.sh`, `templates/*.md`) so each skill is self-contained, and update all internal cross-references (command↔command, command→prompt/script/template, and doc-level references in `README.md`, `quickref.md`, `CLAUDE.md`, `playbook-update.md`). Fold in the Deferred investigation: assess whether the RDPI-artifact restructure can land alongside Task 7 to enable `disable-model-invocation: false`.
- **Constraints:** (a) Skills default to `disable-model-invocation: true` *(verbatim from `tasks/todo.md:9` — **research found this factually wrong: the real default is `false`**; see Summary CORRECTION)* — workflow skills stay manual-invoke-only unless research finds the artifact restructure feasible to land together. (b) Original `.claude/commands/<name>.md` files stay in place until all skills verified; deletion is a separate out-of-scope cleanup pass. (c) No behavioral changes beyond the mechanical port and reference fixes. (d) Flipping `disable-model-invocation` to `false` is blocked on the artifact restructure unless research shows otherwise. (e) Issue #2 (`</dev/null`) and Task 10 (`run_in_background: true`) discipline survives the port unchanged.
- **Acceptance criteria:** Each skill verified by invoking `/<name>` — behavior identical to the command it replaced, *except* the three authorized changes recorded in Cross-Cutting Constraints (the Axis 3 invocation-policy flag, the Axis 4 injection-syntax conversion, and `codex-review`'s target resolution). Each skill self-contained (supporting files co-located or referenced by a still-resolving path). All cross-references repo-wide remain valid, including `playbook-update.md`'s managed-files list and the doc-accuracy sweep. The artifact surfaces a clear, evidence-backed yes/no on bundling the artifact restructure with Task 7.
- **Relevant paths:** `.claude/commands/*.md` (22 files), `.claude/skills/` (new), `.claude/prompts/*.md`, `.claude/scripts/*.sh`, `templates/*.md`, `CLAUDE.md`, `README.md`, `quickref.md`, `tasks/` artifacts, `.claude/settings.local.json`, `.gitignore`.

## Summary

**The skill port is genuinely mechanical and low-risk.** The official Claude Code docs are explicit: *"Custom commands have been merged into skills. A file at `.claude/commands/deploy.md` and a skill at `.claude/skills/deploy/SKILL.md` both create `/deploy` and work the same way."* Commands and skills share the same body format, the same frontmatter, the same `$ARGUMENTS`/`$N` substitution, and the same shell injection. The port is: create `.claude/skills/<name>/SKILL.md` with the command body (optionally adding frontmatter), relocate/keep supporting assets, and fix references. Behavior carries over unchanged.

**CORRECTION — the task's stated default is wrong, and the port is the fix, not the preservation.** `tasks/todo.md:9` ("Upstream constraints") asserts *"Skills default to `disable-model-invocation: true`"*. This is **factually incorrect**. The official docs frontmatter table states the default is **`false`**, and the invocation table's `(default)` row shows "Claude can invoke: Yes." Because custom commands have been merged into skills, the playbook's 22 bare `.claude/commands/*.md` files — which carry **no frontmatter at all** — currently sit at that permissive default and are therefore **already auto-invocable**: Claude can fire them on its own. The desired manual-only posture (per the `feedback_skill_manual_invocation.md` memory) is *not* the current state. This reframes Task 7: porting to skills is the moment to **add `disable-model-invocation: true`**, which *fixes* the unwanted auto-firing — a desired behavioral change, in mild tension with the task's "no behavioral changes" constraint, and one /design should call out explicitly. (The flag could equally be added to the bare command files — `.claude/commands/*.md` "support the same frontmatter" — but it never was; the port is the natural occasion.)

**The Deferred artifact-restructure investigation resolves to a clear NO — and stronger than Codex framed it.** Codex concluded the restructure is "not cheap" (200+ singleton-artifact reference sites, plus new design decisions on task-dir naming, active-task pointer, scratch/log isolation, checkpoint schema, cleanup, audit). That is correct. But the deeper synthesis finding is that **the restructure is not even a real prerequisite for Task 7's success.** The Deferred item assumes auto-invocation is a desired end state gated only on artifact collisions. It is not: every playbook command is a *side-effect workflow* command (writes artifacts, commits, pushes, runs Codex). The official docs say side-effect workflows like `/commit` and `/deploy` *should* carry `disable-model-invocation: true` precisely so Claude does not auto-decide to run them. So `disable-model-invocation: true` is a strong candidate for the **permanent end state** of the heavyweight/side-effect skills, not a temporary compromise awaiting the restructure — making the restructure low-value, since its only payoff (safe broad auto-invocation) is something these skills arguably should not have anyway. The Deferred item can stay deferred and loses essentially little. (Note: per the Axis 3 decision, only `codex-review` stays auto-invocable; the 21 singleton-artifact-writing / side-effect commands get `disable-model-invocation: true`. The restructure would matter only if those were auto-invoked — which the decision rules out.) /design should record this explicitly rather than treating auto-invocation as deferred-but-wanted.

**The breakage surface is small and well-bounded.** Of the cross-reference inventory, only **14 sites hardcode `.claude/commands/<name>.md` paths** that break on a file move — 4 in `auto-issues.md` (child-process `claude -p "Read .claude/commands/issue-*.md"`), 8 in `playbook-setup.md` (global install of commit/push-pr/push-pr-light/catchup), 2 prose convention-references in `codex-review.md`. Everything else is either a `/slash-name` reference (survives, since `/name` invokes the skill identically), a managed-files list to rewrite, or a **doc-level structure/terminology reference** — e.g. `README.md:108` documents a `.claude/commands/*` layout row, and `README.md`/`quickref.md` describe the surface as "commands." Those keep *working* but become *stale documentation*, so the port includes a doc-accuracy sweep (see Cross-Cutting Constraints). The ~200+ singleton-artifact references are *not* touched by the port — they only matter to the (deferred) restructure.

**One genuine behavioral hazard:** five commands — `commit`, `finish`, `issue-finish`, `push-pr`, `push-pr-light` — use bare `!git …` lines (`!git status`, `!git diff`, `!git log`) to preload git output. The documented skill shell-injection syntax is `` !`git status` `` (backtick-wrapped) or fenced ```` ```! ```` blocks. Bare `!command` is undocumented; if it silently renders as literal text in a skill, those commands lose their preloaded diff. This needs an explicit decision in /design and empirical verification in /implement.

`.claude/commands/` holds 22 prompt files; `.claude/skills/` does not yet exist. By family:

- **RDPI workflow:** `research-codebase.md`, `design.md`, `create-plan.md`, `implement.md`, `implement-codex.md`, `create-todo.md`, `finish.md`.
- **Issue workflow:** `issue-research.md`, `issue-plan.md`, `issue-implement.md`, `issue-update.md`, `issue-finish.md`, `auto-issues.md`.
- **Utilities:** `commit.md`, `push-pr.md`, `push-pr-light.md`, `catchup.md`, `checkpoint.md`, `codex-review.md`.
- **Playbook maintenance:** `playbook-setup.md`, `playbook-update.md`, `playbook-audit.md`.

All 22 are **plain Markdown with no YAML frontmatter** — they start directly with `# Title`. Confirmed by sampling `research-codebase.md:1`, `design.md:1`, `implement.md:1`, `issue-research.md:1`, `auto-issues.md:1`, `commit.md:1`, `finish.md:1`, `create-todo.md:1`.

### Command/skill equivalence (official docs)

Per `https://code.claude.com/docs/en/skills`: project skills live at `.claude/skills/<skill-name>/SKILL.md`. *"A file at `.claude/commands/deploy.md` and a skill at `.claude/skills/deploy/SKILL.md` both create `/deploy` and work the same way. Your existing `.claude/commands/` files keep working."* And: *"Files in `.claude/commands/` still work and support the same frontmatter."* The skill body is the markdown after the frontmatter; it behaves exactly as a command body. This means the port is a relocation + optional-frontmatter-addition, not a rewrite.

**Same-name precedence:** *"if a skill and a command share the same name, the skill takes precedence."* Once `.claude/skills/research-codebase/SKILL.md` exists, `/research-codebase` invokes the skill, not the old command. This is good for the acceptance criterion — `/<name>` verification automatically tests the new skill — and it means leaving the old `.claude/commands/*.md` files in place (per constraint b) is harmless: they are simply shadowed.

### Argument handling

Commands today use the literal token `$ARGUMENTS` (in `catchup`, `checkpoint`, `codex-review`, `issue-research`, `issue-plan`, `issue-implement`, `issue-update`, `issue-finish`, `push-pr`, `push-pr-light`, `research-codebase`). Skills support the **identical** `$ARGUMENTS` substitution, plus `$ARGUMENTS[N]`, `$N` shorthand, and named `$name` args (declared via the `arguments` frontmatter field). If `$ARGUMENTS` is absent but args are passed, Claude Code appends `ARGUMENTS: <value>`. **No change needed** — existing `$ARGUMENTS` usage ports verbatim. Note: `research-codebase.md` etc. are the *current* commands using `$ARGUMENTS`; this very `/research-codebase` invocation already uses the skill-flavored template, evidence the substitution is equivalent.

**How they are referenced matters for "self-contained":** every reference is either (a) a Read-tool instruction with a repo-relative path (`Read the prompt template from .claude/prompts/research-guide.md`), or (b) a `bash .claude/scripts/...sh` invocation. Both resolve from the repo root, which is Claude Code's working directory. They are **not** cwd-fragile and do not strictly require co-location to keep working. The `${CLAUDE_SKILL_DIR}` variable is a **general skill-content substitution** — it expands to the skill's own directory and is usable anywhere in the SKILL.md body, including ordinary `bash` instruction blocks (the docs' own example runs `python3 ${CLAUDE_SKILL_DIR}/scripts/visualize.py` as a plain instruction, not a `` !`…` `` injection). It is the cwd-independent way to reference assets *co-located inside a skill directory*. The playbook's current references don't need it — they are repo-relative and resolve from repo root either way — but it is the right tool **if** Axis 2 chooses co-location. So "self-contained" here is an *organizational* goal, not a correctness requirement: repo-relative paths keep working regardless of where SKILL.md sits.

### Shell injection — the one real behavioral hazard

`commit.md:3-4`, `finish.md:5-6` use bare lines `!git status` / `!git diff` to preload git output into the prompt. `push-pr.md:3-4`, `push-pr-light.md:3-4`, `issue-finish.md:8-9` similarly preload state (`push-pr`/`push-pr-light` via `!git log --oneline -3`). `catchup.md` does **not** use injection — it runs git as explicit steps, so it is unaffected. The documented skill syntax (docs §"Inject dynamic context") is `` !`<command>` `` **inline with backticks** (recognized only at line start or after whitespace) or a fenced ```` ```! ```` block for multi-line. Bare `!git status` without backticks is **not documented**. It may be legacy-supported (the commands currently work), but a "no behavioral change" port cannot assume that — if a skill renders bare `!git status` as literal text, `commit`/`finish`/`push-pr`/etc. silently lose their preloaded diff and operate blind. **Decision required (Axis 4).**

### Child-command handoff in /auto-issues

`auto-issues.md` lines 53, 65, 77, 89 spawn child Claude processes whose prompts literally say `Read .claude/commands/issue-research.md and follow its instructions` (and `issue-plan`, `issue-implement`, `issue-update`). Line 77 references the path twice (the structural-mismatch clause). These **4 hardcoded paths break** if the issue-* command files are ever deleted — and even before deletion, they would point at the old command body, not the ported skill, causing the pipeline to silently run stale logic. This is the single most important reference-update site.

### playbook-update.md managed-files list

`playbook-update.md` lines **9–47** contain the canonical "Managed files" list — a fenced block (lines 14–46) of 33 file paths the playbook owns and overwrites wholesale on update. It enumerates all 22 command files plus the 3 prompts, 2 scripts, 5 templates, and `quickref.md`. After the port this list must be rewritten to skill paths (`.claude/skills/<name>/SKILL.md`) or the next `/playbook-update` will manage stale paths.

### playbook-setup.md global install

`playbook-setup.md` lines ~96–114 offer to install four utility commands (`commit`, `push-pr`, `push-pr-light`, `catchup`) into `~/.claude/commands/`. Post-port these should install to `~/.claude/skills/<name>/SKILL.md` (the docs confirm `~/.claude/skills/` is the personal-skill location). This is a decision point, not a forced change — commands keep working — but leaving it installs the old format.

## Architecture Analysis

The playbook is a **prompt-as-code toolkit**: each command is a Markdown procedure the agent executes. There is no compiled artifact, no runtime — the "architecture" is the directory layout plus a dense web of textual cross-references. State flows through disk artifacts under `tasks/` (singleton RDPI files, issue-scoped files, `codex-*.tmp` scratch files, `logs/`). The playbook is also **installed into consuming projects** (`README.md` Setup: `mv playbook/* playbook/.claude .`), so the directory layout is a public contract that `playbook-update.md` and `playbook-setup.md` depend on.

Skills are the strictly-more-capable successor to commands: same body semantics, plus a directory for bundled files, structured frontmatter, optional model-invocation, per-skill tool pre-approval, and subagent execution (`context: fork`). Porting is an upgrade in packaging only. The reason the playbook hasn't already adopted skills is sequencing — Task 7 was deliberately placed last so the port operates on a settled command surface (Tasks 1–6, 8–12 are all complete).

The singleton-artifact design (`tasks/research-codebase.md`, `tasks/plan.md`, …) is a deliberate "one active RDPI task at a time" contract, enforced by hard-stop guards (`research-codebase.md:17-19`) and by `CLAUDE.md`'s workflow + session-start rules. The issue flow already demonstrates the scoped alternative — `tasks/research-issue-<N>.md`, branch-derived active issue in `worktree-issue-<N>`, cleanup in `/issue-finish`. That issue flow is the closest precedent for what an artifact restructure would look like.

## Design Axes

### Axis 1: Skill body shape

- **Choices:** (a) **Direct port** — `SKILL.md` = optional frontmatter + the command body verbatim. (b) **Wrapper** — thin `SKILL.md` that points at a bundled `command.md` holding the body. (c) **Transitional wrapper** — `SKILL.md` instructs "read `.claude/commands/<name>.md`".
- **Per-axis constraints:** Each command must end up as a `.claude/skills/<name>/SKILL.md`. Bodies must port without behavioral change. Docs say command and skill bodies "work the same way," so (a) is fully sufficient.
- **Evidence:** Docs §"Where skills live"; `implement-codex.md` is ~499 lines (`ls` shows 38 KB) — with frontmatter it exceeds the docs' *"Keep SKILL.md under 500 lines"* Tip (a soft guideline, not a hard limit).

### Axis 2: Supporting-asset locality

- **Choices:** (a) **Co-locate per skill** — copy each asset into every skill dir that uses it. (b) **Shared support directory** — keep a shared location for multi-skill assets, co-locate single-use ones. (c) **Keep root layout** — leave `.claude/prompts/`, `.claude/scripts/`, `templates/` exactly where they are.
- **Per-axis constraints:** `codex-output-check.sh` is used by ~10 skills and `error-report.md` / `research-guide.md` by multiple — duplication means N copies to keep in sync, which `playbook-update.md` must then manage. `templates/new-issues.md` and `templates/playbook-sections.md` are referenced by `CLAUDE.md` and `playbook-setup.md`, i.e. outside any single skill, so `templates/` cannot fully collapse into skill dirs. Repo-relative reference paths keep resolving regardless of choice (see "How they are referenced").
- **Evidence:** `codex-output-check.sh` invocation count (~10 commands, inventory Category 4); `CLAUDE.md:28,30` references `templates/new-issues.md`; `${CLAUDE_SKILL_DIR}` is a general skill-content substitution — the cwd-independent way to reference co-located assets — relevant if co-location (choices a/b) is chosen.

### Axis 3: Invocation policy (frontmatter)

- **Choices:** (a) **All skills `disable-model-invocation: true`** (manual-invoke only). (b) **Split by weight** — heavyweight/stateful/side-effect commands get `true`; genuinely lightweight/advisory skills stay auto. (c) **Keep auto-invocation broadly on** (current behavior) and shape misfire propensity via narrow `description` / `when_to_use` / `paths` frontmatter. (d) **Full auto-invocation** (only viable after the artifact restructure — see Axis 6).
- **Per-axis constraints:** **Current state:** the 22 bare command files have no frontmatter and so sit at the default `disable-model-invocation: false` — they are *already auto-invocable* (see the CORRECTION in Summary; `tasks/todo.md:9`'s claim of a `true` default is wrong). So choices (a)/(b) are active changes that *add* the flag; choice (c) preserves the status quo. **Propensity, not a hard gate:** `description`, `when_to_use`, and `paths` shape *how often* Claude auto-fires a skill but cannot express a conditional "fire unless the developer wants a different approach" rule — auto-invocation is a model judgment. The only reliable developer-controlled gate is manual `/command` invocation (i.e. `disable-model-invocation: true`). **Suggest-vs-fire nuance:** `disable-model-invocation: true` removes the skill's `description` from context, so Claude can no longer *suggest* the command either — but the playbook already encodes the RDPI workflow in `CLAUDE.md`, so Claude keeps suggesting the right command from `CLAUDE.md` guidance while `true` only blocks unasked execution. This decouples "Claude recommends the command" from "Claude runs it." Docs explicitly recommend `disable-model-invocation: true` for side-effect workflows (`/commit`, `/deploy`); every playbook command has side effects (writes artifacts, commits, pushes, runs Codex). The cost asymmetry favors `true` for heavyweight commands: a wrong auto-fire spends Codex credits / clobbers a singleton artifact / makes an unwanted commit, whereas a missed auto-fire costs only a keystroke.
- **Decision (developer, 2026-05-21):** resolved to choice (b), split by weight. **`codex-review` ships with auto-invocation enabled** (`disable-model-invocation` omitted or `false`) — it is the only command with no persistent side effects (no commits, no artifacts, self-cleaning tmp file; the cost of a wrong auto-fire is only Codex credits, which the playbook wants used more). **The other 21 commands ship with `disable-model-invocation: true`** — each writes a `tasks/*` artifact, commits, pushes, runs a long Codex sweep, or modifies config, so a wrong auto-fire is the expensive/destructive failure mode. **The issue workflow is unaffected by turning the issue-* commands off:** `/auto-issues` chains its phases by spawning child `claude -p` processes with explicit `Read <file> and follow its instructions` directives (`auto-issues.md:53,65,77,89`), not by model auto-invocation. `disable-model-invocation: true` blocks Claude from invoking the skill at all — both model auto-invocation **and** programmatic (Skill-tool) invocation (docs: *"Use `disable-model-invocation: true` to block programmatic invocation"*). It does **not** block an explicit `Read` of the skill file, nor a human typing `/command` interactively. `/auto-issues` chains via explicit file reads (`Read <file> and follow its instructions`) — a file read, not an invocation — so it is unaffected regardless of the flag. Manual issue-flow chaining works via each command's end-of-body handoff text, same as RDPI — auto-invocation was never the chaining mechanism. /design carries this as a fixed input.
- **Decision (developer, 2026-05-21) — `codex-review` auto-invocation behavior:** because `codex-review` is the one auto-invocable skill, its target resolution must work without an explicit `$ARGUMENTS` (Claude auto-fires it mid-conversation, e.g. the developer says *"…then have codex review"*). The skill gets two entry modes: **(i)** explicit target supplied (`/codex-review foo.ts`) → review that, unchanged; **(ii)** no explicit target (auto-fired, or bare `/codex-review`) → infer the target from conversational context (the work just done, or whatever the chat instruction tagged); if genuinely ambiguous, **ask** a clarifying question (*"review all of this, or just X?"*) rather than guessing or hard-stopping. This **supersedes `codex-review.md:5`'s current hard-stop** ("if no target, stop and tell the developer to re-invoke") for the auto/conversational path — a behavioral change to `codex-review`, authorized as an intended consequence of making it auto-invocable (auto-on is hollow if it hard-stops on every auto-fire). Self-contained: no command invokes `codex-review` (verified by grep), so blast radius is zero. The `description`/`when_to_use` should be **natural, not tight** — fire on phrasings like "have codex review", "get codex's opinion", "codex review this".
- **Evidence:** Docs §"Control who invokes a skill" + invocation table + §"Inject dynamic context" (`paths`, `when_to_use`); `CLAUDE.md` RDPI workflow rules; `feedback_skill_manual_invocation.md` memory.

### Axis 4: Shell-injection syntax for preloaded git state

- **Choices:** (a) **Convert** bare `!git status` → documented `` !`git status` `` inline form (and multi-line groups → ```` ```! ```` fenced blocks). (b) **Keep bare `!git` lines** unchanged and rely on legacy support. (c) **Replace injection with explicit instructions** ("run `git status` and review the output").
- **Per-axis constraints:** Affects exactly five commands — `commit` (`:3-4`), `finish` (`:5-6`), `issue-finish` (`:8-9`), `push-pr` (`:3-4`), `push-pr-light` (`:3-4`); `grep '^!' .claude/commands/` confirms no other file uses injection. The documented forms are `` !`command` `` and ```` ```! ````; bare `!command` is undocumented. Choice (c) is a behavioral change (output arrives via a tool call mid-skill rather than preloaded) — at the edge of the "no behavioral change" constraint.
- **Decision (developer, 2026-05-21):** resolved to choice (a) — convert every bare `!git …` line to the documented inline form `` !`git …` `` (multi-line groups → ```` ```! ```` fenced blocks). It is spec-compliant, a near-zero diff, and removes the undocumented-syntax uncertainty.
- **Evidence:** Docs §"Inject dynamic context" ("The inline form is only recognized when `!` appears at the start of a line or immediately after whitespace" — and the documented inline form is `` !`<command>` ``); `commit.md:3-4`, `finish.md:5-6`.

### Axis 5: /auto-issues child-command handoff

- **Choices:** (a) **Repoint to skill files** — child prompt reads `.claude/skills/issue-research/SKILL.md`. (b) **Invoke by slash** — child prompt says "run `/issue-research <N>`". (c) **Keep reading `.claude/commands/issue-*.md`**.
- **Per-axis constraints:** Choice (c) means the pipeline runs the *old* command body, not the ported skill, and blocks eventual command deletion. Choice (b) is **not viable for the 21 manual-only skills** — `disable-model-invocation: true` blocks programmatic invocation, so a child told "run `/issue-research N`" cannot invoke the skill (see Axis 3 / External Research). Choice (a) — an explicit `Read` of the skill file — is a plain file read, unaffected by the flag, and is exactly what `/auto-issues` does today. Affects `auto-issues.md:53,65,77,89`.
- **Decision (developer, 2026-05-21):** resolved to choice (a) — repoint each child-spawn directive to `Read .claude/skills/issue-<phase>/SKILL.md and follow its instructions`. Choice (b) is ruled out (programmatic invocation of a manual-only skill is blocked); choice (c) is ruled out (runs stale command bodies, blocks deletion).
- **Evidence:** `auto-issues.md:53,65,77,89` child-process invocations; docs §"Control who invokes a skill" (programmatic invocation blocked by `disable-model-invocation: true`).

## External Research

Source: official Claude Code skills documentation — `https://code.claude.com/docs/en/skills` (fetched and verified 2026-05-21).

- **Skill location & equivalence:** project skills at `.claude/skills/<skill-name>/SKILL.md`; *"Custom commands have been merged into skills … both create `/deploy` and work the same way"*; *"Files in `.claude/commands/` still work and support the same frontmatter."* **Unblocks Axis 1** (direct port is sufficient and intended).
- **Frontmatter fields** (all optional, only `description` recommended): `name`, `description`, `when_to_use`, `argument-hint`, `arguments`, `disable-model-invocation`, `user-invocable`, `allowed-tools`, `model`, `effort`, `context`, `agent`, `hooks`, `paths`, `shell`. **Unblocks Axis 3** (frontmatter content).
- **`disable-model-invocation` defaults to `false`** (docs frontmatter table: *"Default: `false`"*; invocation table `(default)` row: "Claude can invoke: Yes"). Setting it to `true` → user-invocable only, description **not** loaded into context, *"Use for workflows with side effects … like `/commit`, `/deploy`."* It also **blocks programmatic invocation** (docs: *"Use `disable-model-invocation: true` to block programmatic invocation"*) — Claude cannot invoke the skill spontaneously *or* when instructed to; only an interactive user typing `/command` and explicit file `Read`s remain unaffected. `user-invocable: false` → Claude-only (background knowledge). **This directly contradicts `tasks/todo.md:9`**, which claims skills default to `true`. Bare command files (no frontmatter) inherit the `false` default and are auto-invocable today. **Unblocks Axis 3** (the 21 manual-only skills must *explicitly set* `disable-model-invocation: true` — `codex-review` excepted, it stays auto-invocable; this is the correct permanent posture, and a change from the current auto-invocable state, not a default).
- **Arguments:** `$ARGUMENTS`, `$ARGUMENTS[N]`, `$N`, `$name` (via `arguments` frontmatter) all supported; missing-`$ARGUMENTS` fallback appends `ARGUMENTS: <value>`. **Unblocks Axis 1** (existing `$ARGUMENTS` ports verbatim).
- **`${CLAUDE_SKILL_DIR}`** expands to the skill's directory; it is a general skill-content substitution usable anywhere in the SKILL.md body (the docs use it in a plain `bash` instruction block, not only in `` !`…` `` injection). Other vars: `${CLAUDE_SESSION_ID}`, `${CLAUDE_EFFORT}`. **Unblocks Axis 2** — it is the cwd-independent way to reference assets co-located in a skill directory; the playbook's current repo-relative references resolve from repo root either way, so co-location is organizational rather than a correctness requirement, but `${CLAUDE_SKILL_DIR}` is the clean mechanism if co-location is chosen.
- **Shell injection:** documented forms are inline `` !`command` `` (recognized only at line start or after whitespace; literal if `!` follows another char) and fenced ```` ```! ```` blocks. Bare `!command` (no backticks) is **not documented**. **Unblocks Axis 4** — converting bare `!git status` to `` !`git status` `` is the safe, spec-compliant choice.
- **Same-name precedence:** *"if a skill and a command share the same name, the skill takes precedence."* **Unblocks Axis 5** and the verification approach — `/<name>` tests the new skill once the skill exists; old command files coexist harmlessly.
- **Live change detection:** edits to existing skill dirs apply mid-session; **creating the top-level `.claude/skills/` dir requires a Claude Code restart** before discovery. Confirms the verification-timing risk below.
- **SKILL.md size:** *"Keep SKILL.md under 500 lines"* is a Tip, not a hard limit. `implement-codex.md` (~499 lines) is the only file near it.
- **`disableSkillShellExecution` setting** can globally disable `` !`command` `` / ```` ```! ```` injection (replacing it with a placeholder) — out of scope but a known kill-switch for Axis 4 behavior.

No contradictions between the docs and the codebase were found. Codex's external claims (`.claude/skills/<name>/SKILL.md` layout, frontmatter field list, `${CLAUDE_SKILL_DIR}`, the `allowed-tools`-pre-approves-not-restricts correction, the restart-for-new-dir requirement, the bare-`!` correction) all hold up against the official docs.

## Skill-Feature Adoption Audit

Task 7 is scoped as a mechanical port, but skills expose capabilities plain commands lack. This audit classifies each by fit, so /design can decide what (if anything) to adopt rather than leaving it implicit. Source: official skills docs (see External Research).

**Tier 1 — zero behavioral risk, fold into the port. DECISION (developer, 2026-05-21): adopt both.**
- `argument-hint` — autocomplete hint (e.g. `[issue-number]`, `[target]`). UX only; no execution change. Applies to the ~11 arg-taking commands (`catchup`, `checkpoint`, `codex-review`, `issue-*`, `push-pr`, `push-pr-light`, `research-codebase`).
- `description` (and `when_to_use` for `codex-review`) — improves the `/` menu; load-bearing for `codex-review`'s auto-invocation matching (per the Axis 3 decision). Effectively table stakes for a well-formed skill.

**Tier 2 — genuine value, but a behavioral change. DECISION (developer, 2026-05-21): do not adopt.**
- `allowed-tools` — pre-approves tools so a skill runs without permission prompts; docs show a `commit` skill with `allowed-tools: Bash(git add *) Bash(git commit *)`. High value for git-heavy skills (`commit`, `push-pr`, `push-pr-light`, `catchup`, `finish`, `issue-finish`). Caveats: it alters the permission surface (a behavioral change vs. constraint c), interacts with the existing `.claude/settings.local.json` allow-list and the `skipAutoPermissionPrompt` flag, and project-skill `allowed-tools` only takes effect after the workspace-trust dialog is accepted. Skipped for Task 7: the permission surface is already owned by the per-machine `.claude/settings.local.json` allow-list (which already pre-approves `Bash(git *)`, `Bash(gh pr *)`, `Bash(codex *)`, etc.), and adding *committed* `allowed-tools` would be a posture change — shipping tool pre-approval into every consuming project — that a mechanical port does not need. (For the record: invoking a tool "directly as a skill step" does **not** itself suppress the permission prompt — only an allow-list or `allowed-tools` does — so this skip rests on the existing settings allow-list, not on direct invocation.)

**Tier 3 — do not adopt (wrong fit or out of scope):**
- `context: fork` + `agent` — runs the skill as an isolated subagent with no conversation history. Fundamentally incompatible with the RDPI commands, which depend on Claude synthesizing *with* full context inline. The playbook already spawns sub-agents deliberately *within* commands where isolation is wanted; making whole commands forked would break the synthesis model.
- `effort` / `model` per-skill overrides — behavioral, no demonstrated need.
- `hooks` (skill-scoped lifecycle hooks) — could enforce `</dev/null` / output-check discipline, but it is a separate enforcement mechanism warranting its own design; out of scope for a port.
- `arguments` (named positional `$name` substitution) — a body rewrite for marginal gain over flat `$ARGUMENTS`, which the commands already use and which ports verbatim.
- `paths` — glob-gated auto-activation; moot for the 21 manual-invoke skills.

## Risk Analysis

- **Verification timing (highest-impact).** Creating `.claude/skills/` for the first time requires a Claude Code restart before any skill is discoverable. The acceptance criterion "verify each skill by invoking `/<name>`" therefore cannot complete in the same session that first creates the directory. The implement phase must account for a restart (or a session boundary) between "skills written" and "skills verified" — this is a real sequencing constraint, not a nicety.
- **Bare-`!` shell injection silently degrading.** If bare `!git status` renders as literal text in a skill, `commit`/`finish`/`push-pr`/`push-pr-light`/`issue-finish` lose their preloaded diff and act on no/stale state — a silent correctness failure. Must be resolved (Axis 4) and empirically verified.
- **/auto-issues running stale logic.** Until `auto-issues.md:53/65/77/89` are repointed, the pipeline's `claude -p` children read the old `.claude/commands/issue-*.md` bodies, not the ported skills — divergence that won't surface until a full pipeline run.
- **Half-ported skill shadows a working command.** Same-name precedence means a broken/incomplete `SKILL.md` immediately shadows the functioning command. Port-and-verify each skill before moving on; do not bulk-create skeletons.
- **`playbook-update.md` managed-files drift.** If the managed-files list isn't rewritten in lockstep, the next `/playbook-update` manages stale paths and may fail to update or may resurrect deleted commands.
- **Discipline regression.** `</dev/null` and `run_in_background: true` are textual conventions with no lint. A careless copy/reformat during the port could drop them from a long-running Codex/`claude -p` site. Spot-check every such site post-port.
- **Asset duplication drift (if Axis 2 = a).** Duplicated `codex-output-check.sh` / `research-guide.md` copies fall out of sync over time; `/playbook-update` would need to manage each copy.
